Last week <a title="Alex's blog" href="https://blogs.igalia.com/alex/">some</a> <a title="My blog :P" href="/">of</a> <a title="Martin's blog" href="http://blog.abandonedwig.info/">the</a> <a title="Phil's blog" href="http://base-art.net/">members</a> <a title="Sergio's blog" href="https://blogs.igalia.com/svillar/">of</a> <a title="Xan's blog" href="http://blogs.gnome.org/xan/">the</a> <a title="Igalia" href="http://www.igalia.com">Igalia</a> <a title="The WebKit Project" href="http://www.webkit.org">WebKit</a> <a title="WebKitGTK at igalia.com" href="http://www.igalia.com/nc/work/project/item/webkitgtk/">team</a>,  attended to the <a title="http://trac.webkit.org/wiki/April%202011%20Meeting" href="http://trac.webkit.org/wiki/April%202011%20Meeting">second edition of the WebKit Contributors Meeting</a> in Cupertino, California, in order to gather round with other <a title="The WebKit Project" href="http://www.webkit.org">WebKit</a> contributors to discuss and work around our favorite web engine, away from IRC and with a more personal and "human" touch.

<a href="http://www.flickr.com/photos/torarnv/5684090280"><img class="aligncenter" src="/wp-content/uploads/2011/05/20110505-wkcm.jpg" alt="WebKit Contributors Meeting '11 - group photo" width="640" /></a>

As for me, it was the first time I attended this unconference and I have to say it was a great experience overall, even if accessibility (the field I usually work on through <a title="WebKitGTK" href="http://www.webkitgtk.org">the GTK port of WebKit</a>) was not precisely a hot topic there. But in the other hand, I managed to put some faces to people I just knew from IRC, attended to several interesting discussions and sessions, did some actual work (tm) and met Chris Fleizach, the guy from Apple involved in accessibility for the Mac port, who attended the meeting on Tuesday morning and discussed with me some interesting topics, mainly about the implementation of the accessibility support in the Mac port of <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a>.

For those of you that are not up to date with what <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a> means and what's the current state of the whole thing, I'd recommend you to take a look to the <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2 wiki</a>, although I can already advance you that one of the main features (if not the main one) of <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a> is the new <strong>multi-process architecture,</strong> which will go bundled right along with the engine, instead of doing the split in the final application, like Google Chrome does now with <a title="The WebKit Project" href="http://www.webkit.org">WebKit1</a> (see the wiki for more details, and some nice diagrams too, like the following one).

<a href="http://trac.webkit.org/wiki/WebKit2"><img class="aligncenter" src="/wp-content/uploads/2011/05/wk2-stack.png" alt="WebKit2 multi-process architecture" width="359" /></a>

The problem however, from the accessibility point of view, is how to expose such a new architecture to Assistive Technologies (<em>ATs</em>) while, at the same time, having those <em>ATs </em>seeing just one process (the <em>UI process</em>), regardless of the different processes that would be running on each of the tabs (the <em>Web proceses</em>). In other words, how to "hide" to the <em>ATs </em>that they are extracting accessibility related information from a multi-process based browser, so they still keep the illusion of having only one application exposing accessibility objects, as it used to be so far.

Talking to Chris about how they implemented this in the Mac port, he told me they basically needed to make up a way to transparently connect the <em>UI process</em> and the <em>Web Process</em>, so navigation between the root accessibility object in the <em>Web Process </em>and the "leaf" accessibility object in the <em>UI process</em> were done in a seamless way, through a mechanism that would basically allow bidirectional communication in the same way it used to be when there was just one process running. This makes a lot of sense, if you ask me, since you "just" (saying it as if it were not a complex task, even though it actually is) add this mechanism to <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a> accessibility code and you'll <em>automagically </em>get your <em>ATs </em>working as they used to work, without any other changes needed from their side.

However, when it comes to <a title="ATK/AT-SPI" href="http://www.linuxfoundation.org/collaborate/workgroups/accessibility/atk/at-spi">ATK/AT-SPI</a>, which is what we currently use in <a title="The GNOME Project" href="http://www.gnome.org">GNOME</a> to expose accessible objects from applications to <em>ATs</em>, it looks like such a mechanism (or something pretty similar) is already available by means of the <a title="AtkSocket documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkSocket.html"><em>AtkSocket</em></a> and the <a title="AtkPlug" href="http://developer.gnome.org/atk/unstable/atk-AtkPlug.html"><em>AtkPlug</em></a> classes, both subclasses of <em><a title="AtkObject documentation" href="http://developer.gnome.org/atk/unstable/AtkObject.html">AtkObject</a></em>, which basically act as a bridge that allows to connect two <em><a title="AtkObject documentation" href="http://developer.gnome.org/atk/unstable/AtkObject.html">AtkObject</a>'s</em> so the children of one of those (the <em>Plug</em>) are exposed as children of the other one (the <em>Socket</em>), no matter they're in the same process or in different ones. And this, unless I'm missing something, is exactly what we'd need in <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a>, probably along with some other things and tweaks that I just can't think of at this moment. But at least is definitely a very good start point, IMHO.

But... how to communicate those <em>sockets</em> and <em>plugs</em> if they are in different processes? You might be wondering... Well, if you are using the <a title="AT-SPI2" href="http://www.linuxfoundation.org/collaborate/workgroups/accessibility/atk/at-spi/at-spi_on_d-bus">D-Bus implementation of AT-SPI</a>, also known as AT-SPI2, the needed bits for that (the implementation for the <em><a title="atk_socket_embed() documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkSocket.html#atk-socket-embed">atk_socket_embed()</a></em> and <em><a title="atk_plug_get_id() documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkPlug.html#atk-plug-get-id">atk_plug_get_id()</a> </em>functions) are already implemented by the ATK bridge <a title="ATK bridge for AT-SPI2" href="http://git.gnome.org/browse/at-spi2-atk">(<em>at-spi2-atk</em>)</a>, so the only thing you'll need to do is to provide the remaining implementation of the <em><a title="AtkSocket documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkSocket.html">AtkSocket</a></em> and <a title="AtkPlug" href="http://developer.gnome.org/atk/unstable/atk-AtkPlug.html"><em>AtkPlug</em></a><em> </em>classes, register those new <em><em><a title="AtkObject documentation" href="http://developer.gnome.org/atk/unstable/AtkObject.html">AtkObject</a>'s</em></em> as the right accessible objects for your widgets (or whatever your "normal" objects are) and to use the ID that <a title="ATK bridge for AT-SPI2" href="http://git.gnome.org/browse/at-spi2-atk"><em>at-spi2-atk</em></a> provides for the <em><a title="AtkPlug" href="http://developer.gnome.org/atk/unstable/atk-AtkPlug.html">AtkPlug</a></em> object to connect from the <em><a title="AtkSocket documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkSocket.html">AtkSocket</a></em> object... and you're done.

However, I have to say that the documentation I've found so far about <em><a title="AtkSocket documentation" href="http://developer.gnome.org/atk/unstable/atk-AtkSocket.html">AtkSocket</a> </em>and <em><a title="AtkPlug" href="http://developer.gnome.org/atk/unstable/atk-AtkPlug.html">AtkPlug</a></em> is not precisely very detailed, so I basically ended up looking directly at the code and trying to write myself a small example to better understand things. And at this point, I asked in the #a11y channel in GIMPNet and it was Mike Gorse who kindly handed me out a nice tarball with an example they had written in <em>C#</em>, which was exactly what I was looking for. Yay! Thanks, Mike!

However, I'd be more interested in a plain C, GObject based, implementation of that example which would serve me both to better understand how it works and to use it as the base for further tests in <a title="WebKitGTK" href="http://www.webkitgtk.org">WebKitGTK</a>, so I went ahead and wrote it, and this is how the results look now, as seen through <a title="Accerciser" href="https://live.gnome.org/Accerciser">Accerciser</a>'s eyes:

<a href="/wp-content/uploads/2011/05/20110505-atksocket-atkplug.png"><img class="aligncenter" src="/wp-content/uploads/2011/05/20110505-atksocket-atkplug.png" alt="AtkSocket / AtkPlug example in action" width="640" /></a>

As you can see, there are two processes running, <em>ta-socket</em> and <em>ta-plug</em>, and in <em>ta-socket</em> we have an <em><em><a title="AtkObject documentation" href="http://developer.gnome.org/atk/unstable/AtkObject.html">AtkObject</a></em></em> named "The Socket" which is exposing, as his only child, the whole subtree present in the other process <em>ta-plug</em>, starting in another <em><em><a title="AtkObject documentation" href="http://developer.gnome.org/atk/unstable/AtkObject.html">AtkObject</a></em></em> named "The Plug"... which is exactly, although of course at a much smaller dimension, what I think we would like to see happening in the accessibility tree of any <a title="WebKitGTK" href="http://www.webkitgtk.org">WebKitGTK</a> based application in the future, once that <a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a> is mature enough to replace <a title="The WebKit Project" href="http://www.webkit.org">WebKit1</a>.

Obviously, this shouldn't be taken as a "we're done with <a title="WebKitGTK" href="http://www.webkitgtk.org">WebKitGTK</a>/<a title="WebKit2 - High Level Document" href="http://trac.webkit.org/wiki/WebKit2">WebKit2</a> when it comes to a11y" comment or the like. Actually this is just the beginning of the whole thing... But I think, or at least I would like to think, that is at least a good start point :-).

And before you shoot... "yes", I'm already planning to help improve the <a title="ATK Documentation" href="http://developer.gnome.org/atk/unstable/atk.html">ATK documentation</a> in this regard with the knowledge I acquired while working on this examples, even perhaps next week during the <a title="ATK and AT-SPI Hackfest 2011" href="https://live.gnome.org/Hackfests/ATK2011">ATK/AT-SPI hackfest</a> (it's actually one of the <a title="ATK/AT-SPI hackfest proposed tasks" href="https://live.gnome.org/Hackfests/ATK2011#Proposed_Tasks">proposed tasks</a>) I'll be attending here in Coruña, at the <a title="Igalia" href="http://www.igalia.com">Igalia</a> offices.
